# OpenClaw Web Positioning

## 1. 这网站到底是干什么的

本项目网页不是官网，不是普通管理后台，也不是单纯的 3D 展示页。

它的核心定位是：

- `OpenClaw` 在“智能巡拢家居机器人”项目里的落地载体
- 机器人上层任务操作台
- 状态可视化与调试平台
- 比赛答辩和产品演示界面

一句话说：

**这个网站存在的目的，就是把 `OpenClaw` 的能力接到你的低矮寻物取物机器人上，让机器人从“会动、会夹”升级成“会理解任务、会解释动作、会闭环完成取物”的系统。**

## 2. 对当前机器人项目的意义

你的机器人不是扫地机器人路线，而是：

- 面向床底、沙发底、柜底等低矮黑暗空间
- 专门执行遗落物发现、识别、靠近、夹取、退出

因此网页的意义，不是辅助控制，而是把以下能力收拢成一个统一入口：

- 找什么
- 怎么找
- 什么时候夹
- 夹完怎么退
- 失败后怎么重试
- 当前为什么这样做

没有这个网页，`OpenClaw` 只能停留在命令行或后端接口层；
有了这个网页，`OpenClaw` 才真正变成机器人可用的人机交互层。

## 3. OpenClaw 在网页里的角色

第一阶段，不建议把 `OpenClaw` 理解成“直接控制每个电机”的底层系统。

对你当前机器人，更合理的角色是：

- 高层任务理解
- 操作策略生成
- 任务阶段切换
- 动作原因解释
- 多轮交互记忆

推荐链路：

```text
用户网页输入任务
    -> OpenClaw 理解任务
    -> 生成执行策略
    -> 上位机拆成视觉/运动/夹持步骤
    -> STM32 执行动作
    -> 网页实时显示状态与结果
```

## 4. 网页必须承担的功能

### 4.1 任务入口

网页需要有一个明确的任务输入区，例如：

- “帮我找沙发底下的遥控器”
- “优先找钥匙，找不到再找玩具”
- “夹到之后退出来停在我脚边”

这部分是 `OpenClaw` 的直接入口。

### 4.2 决策解释

网页不能只显示“正在执行”，还要显示：

- 当前识别到的目标
- 当前任务阶段
- 当前动作理由
- 为什么减速
- 为什么不夹
- 为什么重新对准

这会把系统从黑箱变成可解释系统。

### 4.3 设备状态

网页需要同时显示：

- 摄像头画面
- 底盘状态
- 夹持机构状态
- 传感器状态
- 串口/通信状态
- 当前任务日志

### 4.4 调试与复盘

网页也是后续调 `OpenClaw` 的实验平台，应支持观察：

- 任务成功率
- 失败案例
- 参数变化
- 决策过程
- 历史任务记录

## 5. 页面结构建议

围绕当前目标，主页面建议分为这几块：

- Header
  - 机器人名称
  - 在线状态
  - 当前任务状态
- Left
  - `OpenClaw` 任务输入
  - 快捷任务按钮
  - 模式切换
- Center
  - 机器人 3D 主视图区
  - 当前设备主视觉
- Right
  - 决策解释
  - 视觉识别结果
  - 传感器/串口状态
- Bottom
  - 日志
  - 历史任务
  - 调试信息

## 6. 为什么要用这些轮子

### 6.1 Arwes

`Arwes` 的作用不是为了“炫”，而是为了把网页做成真正像机器人操作台的 HUD 界面：

- 标题条
- 面板框体
- 状态标签
- 科技感层级

### 6.2 react-three-fiber / drei

这套用来承载：

- 机器人 3D 主视图区
- 模型预览
- 姿态展示
- 后续数字孪生与资产加载

### 6.3 @react-three/postprocessing

这套用来做轻量 3D 视觉增强：

- Bloom
- ToneMapping
- Vignette
- 轻量 AO

目的是让中间主视图区更有沉浸感，但不影响信息可读性。

## 7. 当前图片素材怎么用

当前目录里已经有：

- `robot.jpg`

这张图现在就有实际用途：

- 首页主视觉参考

- 后续 3D 建模与数字孪生参考图

## 8. 当前结论

这个网站的目的非常明确：

- 不是单纯远程控制
- 不是普通 3D 展示
- 不是仅用于好看

而是：

**把 `OpenClaw` 接入你的“智能巡拢家居机器人”后，作为任务理解、状态解释、操作编排、视觉呈现和调试迭代的统一工作台。**
