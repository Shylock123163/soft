# Web Project Plan

## 1. 当前目标

为 `sr` 机器人项目建立一套独立网页前端，用来承载：

- 开屏动画
- 机器人控制台
- 视频/传感器/串口状态展示
- 后续智能助手与 3D 可视化

当前网页项目不是官网展示页，而是偏“机器人控制中台”。

## 2. 当前目录现状

当前 `web-part/web-ui/` 目录包含：

- `app/` — React/Vite/TypeScript 前端主工程（5 页面路由）
- `openclaw-server.js` — OpenClaw 任务后端（Node.js，端口 9012）
- `webhook.js` — GitHub webhook 自动部署
- `workers/r2-assets.js` — Cloudflare Worker 文件管理
- `wrangler.toml` — Worker 配置
- `pm2.ecosystem.config.cjs` — PM2 进程管理
- `deploy.nginx.example.conf` — nginx 示例配置
- `server.env.example` — 环境变量示例
- `robot.jpg` — 机器人外观参考图
- `splash-source/` — www-genshin 开屏参考（未集成）
- 8 个 `.md` 文档

## 3. 当前整体架构思路

### 3.1 系统分层

当前整体思路是三层：

```text
网页前端
    -> 上位机 / 鲁班猫后端
    -> STM32 电控
```

对应关系：

- 网页前端
  - 负责展示、控制、交互、开屏、3D 可视化
- 上位机 / 鲁班猫后端
  - 负责视觉推理、状态整理、串口桥接、接口输出
- STM32 电控
  - 负责底层执行、状态机、运动控制

### 3.2 前端不直接侵入 STM32

网页不应该直接绑死在 STM32 固件层，原因是：

- STM32 更适合做底层控制
- 网页需要更快迭代
- 视觉和智能逻辑更适合放在上位机层

因此网页默认通过：

- HTTP 接口
- 视频流接口
- 状态接口
- 串口桥接后的高层状态

去跟机器人交互。

## 4. 当前前端路线

### 4.1 主前端定位

最终要做的是一套“机器人控制台”式网页，而不是单纯的视频检测页。

更准确地说，这个网站不是官网，也不是普通后台，而是：

- `OpenClaw` 在本项目中的落地载体
- 机器人上层任务中枢
- 状态可视化与调试入口
- 后续比赛展示和产品化演示界面

它的核心目标不是“把机器人放到网页上”，而是：

- 让用户通过网页给机器人下达高层任务
- 让 `OpenClaw` 在网页里承接任务理解与动作编排
- 让底层视觉、底盘、夹持机构的执行过程可见、可解释、可调试

也就是说：

- STM32 负责底层执行
- 上位机负责视觉与桥接
- 网页负责把 `OpenClaw` 变成可用的人机交互层和任务操作台

目标页面应包含：

- Header
  - 设备状态
  - 当前模式
  - 急停 / 启停
- Left Panel
  - 手动控制
  - 参数调整
  - 模式切换
- Center
  - 3D 主视图区
- Right Panel
  - 传感器状态
  - 串口状态
  - AI / 视觉结果
- Bottom
  - 日志 / 最近动作 / 调试信息

### 4.2 技术栈约定

当前前端实际采用：

- 普通 UI：React DOM + 自定义 CSS 设计系统
- 视觉风格：暗色毛玻璃 + 新拟态（两色系统：`rgba(0,0,0,0.6)` + `rgb(234,234,239)`）
- 3D 场景：`@react-three/fiber` + `@react-three/drei`
- 状态管理：`zustand`
- 动画：`framer-motion`
- 图标：`lucide-react`
- 路由：`react-router-dom`

也就是说：

- 控制台部分用普通 React + 自定义暗色毛玻璃 CSS 做
- 3D 区域单独用 `react-three-fiber`
- 动画过渡用 `framer-motion`
- 不把整页都做成纯 3D UI
- 不使用 Arwes 或 postprocessing

### 4.3 网站围绕 OpenClaw 的职责

围绕本项目目标，这个网站应该承担 5 个明确职责：

1. 任务入口
   - 用户在网页里发出“找什么、夹什么、如何处理”的高层指令
2. 状态解释
   - 页面实时展示识别结果、任务阶段、动作理由、失败原因
3. 操作编排
   - 网页把视觉识别、底盘微调、夹持动作和 `OpenClaw` 决策串成闭环
4. 调试面板
   - 用来观察参数、记录日志、分析成功率、回放失败场景
5. 演示载体
   - 比赛/答辩时，把“具身智能取物”完整过程直观地展示出来

因此网页的价值不是附属功能，而是 `OpenClaw` 接入后的主战场。

## 5. 开屏动画思路

### 5.1 当前开屏来源

当前已下载：

- `gamemcu/www-genshin`

本地位置：

- `web-ui/splash-source/`

它的作用不是直接作为最终主项目，而是：

- 提供开屏动画结构参考
- 提供可抽离的启动场景
- 提供全屏沉浸式开场能力

### 5.2 开屏在整体中的角色

开屏动画应该是网页的一个阶段，而不是整站架构基础。

推荐模式：

```text
Splash Screen
    -> Enter
    -> Robot Dashboard
```

即：

- 先播放开屏
- 用户点击进入或自动结束
- 再切到真正的机器人控制台

### 5.3 开屏改造原则

后续改造 `splash-source` 时，原则是：

- 保留可用的动画技术结构
- 去掉不必要的原项目品牌感
- 替换成机器人项目自己的标题、按钮和入口逻辑
- 开屏结束后跳转到机器人控制台

## 6. 与现有 `sr` 仓库的关系

### 6.1 已有后端基础

`sr` 仓库中已经有成熟的上位机网页与桥接基础，尤其是：

- `soft/sweep_bushu (2)(2)/deploy_web_detect.py`

这层已经有：

- Flask 服务
- 视频画面
- 状态输出
- 串口联动 STM32
- 配置文件
- 开机自启动

因此未来网页不是从零做全栈，而是：

- 新前端负责更高级的展示与交互
- 后端优先复用现有 Python 服务能力

### 6.2 前端和后端的推荐关系

推荐关系是：

```text
web-ui/app
    -> 请求 Python 后端接口 / OpenClaw 服务接口
    -> 读取视频流 / 状态 / 控制接口 / 任务结果
    -> 后端再桥接到 STM32
```

这能保证：

- 前端和控制逻辑解耦
- 后续更换页面风格不影响底层
- 后续加入智能助手也更容易

### 6.3 OpenClaw 在系统里的推荐位置

推荐把 `OpenClaw` 放在“上层任务理解与动作决策”位置，而不是直接替代底层控制：

```text
网页任务输入
    -> OpenClaw 任务理解 / 策略生成
    -> 上位机执行编排
    -> STM32 底盘 / 夹持动作执行
    -> 网页回显状态与结果
```

这样更符合你当前硬件形态：

- 你现在是低矮底盘 + 视觉 + V 型夹持机构
- 不是完整多自由度机械臂

所以 `OpenClaw` 的第一阶段作用应聚焦于：

- 理解用户要找什么
- 判断是否值得夹
- 给出接近、对准、闭合、退出的动作策略
- 解释当前动作逻辑

## 8. 图片转 3D 接入思路

### 8.1 当前推荐方案

图片转 3D 不建议直接塞进前端浏览器里做，而是走：

```text
前端上传图片
    -> 自己的后端任务接口
    -> 后端调用 Replicate / 3D 生成服务
    -> 返回 glb/usdz/预览图
    -> 前端 3D 区加载模型
```

原因：

- 前端只负责上传、轮询、展示
- 第三方 token 不暴露在浏览器
- 后续可替换成 Meshy / Tripo / 自建模型服务
- 生成失败、重试、缓存、审核都更好控

### 8.2 当前接入定位

对当前 `sr` 项目来说，图片转 3D 的角色不是“核心控制链路”，而是“资产生成链路”。

它适合放在：

- 开屏后的机器人控制台里
- 单独一个“3D 资产/模型工坊”页签
- 或后续的机器人数字孪生编辑页

它不应该直接干扰：

- STM32 控制
- 串口桥接
- 实时视频
- 机器人状态接口

### 8.3 页面上的落点

后续主控制台建议多一列或一个独立页面：

- `Assets`
  - 上传参考图
  - 生成 3D 模型
  - 查看任务进度
  - 一键载入 3D 视图区
- `Scene`
  - 当前机器人模型
  - 环境模型
  - 生成后的 `glb` 预览

### 8.4 前后端建议接口

建议后续统一用你自己的接口名，不让前端直接感知第三方平台：

- `POST /api/assets/image-to-3d`
  - 上传图片并创建任务
- `GET /api/assets/tasks/:taskId`
  - 查询生成状态
- `GET /api/assets/models/:assetId`
  - 获取模型信息
- `GET /api/assets/models/:assetId/download`
  - 下载 `glb`
- `POST /api/assets/models/:assetId/load`
  - 载入当前 3D 场景

### 8.5 前端显示建议

3D 资产接入后，前端使用：

- `@react-three/fiber`
- `@react-three/drei`

来做：

- `glb` 加载
- 环境光
- 相机控制
- 模型缩放与居中
- 材质和阴影基础预览

不要一开始就做太复杂的建模编辑器，先把：

- 上传
- 生成
- 轮询
- 预览
- 场景挂载

这 5 步跑通。

## 9. 一句话总方案

当前网页项目的整体思路是：

**以 `web-ui/` 作为独立前端工作区，建设一个以 `OpenClaw` 为上层任务中枢的机器人控制台（5 页面多路由架构）；前端负责任务输入、状态解释、3D 可视化与调试交互，采用自定义暗色毛玻璃 CSS + react-three-fiber + framer-motion 技术栈；后端继续复用 `sr` 现有 Python/鲁班猫桥接能力，STM32 保持底层执行角色，图片转 3D 则作为独立资产链路接入到控制台而不是直接侵入底层控制。**
